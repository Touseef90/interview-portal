.NET Interview Question

1. What is .NET Core?
It is a completely new farmework, which is a FREE and OPEN-SOURCE platform developed and maintained by Microsoft.
2. What are the advantages of .NET Core over .NET Framework?
.NET Core provides a faster, lightweight, and cross-platform alternative to the Windows-only .NET Framework. Its main advantages include the ability to run applications on Linux and macOS, significantly faster processing speeds, a modular architecture, and native support for modern containerized cloud deployments.
3. Explain default project structure in ASP .NET Core application?

Root Files and Entry Points

  • Program.cs: The absolute entry point and brain of your application. It initializes the web server, registers services into the dependency injection container, and configures the HTTP request middleware pipeline.
  • ProjectName.csproj: The MSBuild XML file containing project configurations, target framework versions, and explicit NuGet package dependencies.
  • appsettings.json: The central configuration file where you store environment-specific variables like database connection strings, logging thresholds, and API keys.

Core Folders

  • wwwroot: The dedicated web root folder for all public static assets. Any CSS stylesheets, JavaScript files, icons, or images served directly to the browser live here.
  • Properties: Houses the launchSettings.json file. This file dictates local development profiles, including the application URL, target port configurations, and runtime environment settings.
  • Dependencies: Visual Studio node displaying all imported frameworks, SDK references, and NuGet packages your project needs to build.

Model-View-Controller (MVC) Specific Folders

If your app is built using the MVC template.

  • Controllers: Contains C# classes that receive HTTP requests, interact with your business logic, and hand data over to your user interface.
  • Models: Holds the data structures, plain C# objects (POCOs).
  • Views: Contains Razor (.cshtml) front-end templates that mix HTML markup with C# syntax to render data dynamically to clients.
4. What is the role of configureservices and configure method?
In ASP.NET Core, ConfigureServices registers application dependencies into the Dependency Injection (DI) container. It is optional. It gets called by the host before Configure method, while Configure sets up the HTTP request and response pipeline. It is used to add middleware components. These two methods dictate how an application prepares its internal logic and how it handles web requests.
While historically located inside the Startup.cs class, modern .NET applications (using Minimal APIs) execute these exact same responsibilities sequentially using builder.Services and app inside the Program.cs file.
5. What is dependency injection in ASP .NET Core? Or how can we inject the service dependency into the controller?
Dependency Injection (DI) used to achieve loose coupling between classes and their dependencies. Instead of a class directly instantiating its required dependencies (using the new keyword), the ASP.NET Core built-in Inversion of Control (IoC) container automatically creates and provides those instances at runtime.
To inject a service dependency into a controller, you must follow a three-step process: define the service contract, register the service inside the framework's container, and demand the dependency inside your controller.
6. Describe service lifetime in ASP.NET Core. What is the difference between add singleton, add transient and add scoped?
Service lifetime dictates how long a Dependency Injection (DI) instance lives in memory. There are three types:

Transient

  • What it is: A new instance is created every single time it is requested.
  • When to use: Lightweight, stateless services (e.g., utility classes, math helpers).

Scoped

  • What it is: A new instance is created once per HTTP request. It is reused for all calls within that specific request, then destroyed when the request ends.
  • When to use: Services tied to a single user/request (e.g., Entity Framework `DbContext`, Unit of Work).

Singleton

  • What it is: A single instance is created once per application lifetime. It is shared across all users and all requests until the app shuts down.
  • When to use: Heavy-to-create, shared resources (e.g., Memory Cache, Configuration).
7. What is middleware?
A middleware is a component (class) that is executed on every request. When a request comes in, it passes through each middleware component one by one.
8. What is Kestrel? What is in process and out of process hosting?
Kestrel is the default, cross-platform web server for ASP.NET Core applications. It is lightweight, high-performance, and designed to process web requests directly without needing to be bundled with a specific operating system.

In-Process Hosting

  • How it works: IIS acts as the native HTTP server, routing requests directly to the managed code without needing Kestrel.
  • Pros: Maximum performance and lower latency because it eliminates the overhead of proxying requests between two servers.
  • Cons: Windows-only (if relying on IIS), and if the application crashes, the entire IIS worker process goes down

Out-of-Process Hosting

  • How it works: An external web server (like IIS, Nginx, or Apache) acts as a reverse proxy. It listens for incoming traffic and forwards it to the Kestrel process where the app is executing.
  • Pros: Superior isolation and stability—if your ASP.NET Core app crashes, the main reverse proxy stays up. It is also fully cross-platform (works on Linux/macOS).
  • Cons: Slightly slower due to the overhead of proxying requests.
9. What is difference between Kestrel and IIS?
Kestrel is a lightweight, cross-platform, open-source web server designed specifically to run ASP.NET Core applications. In contrast, IIS (Internet Information Services) is a full-featured, enterprise-grade, closed-source web server developed by Microsoft that runs exclusively on Windows.
10. What is Request Delegate? What are Run, Use, and Map methods?
Request Delegate handles each HTTP request and are used to build request pipeline by using the Run, Use, and Map extension methods.
  • The Use method is used to add middleware components to the pipeline. It is designed to run code both before and after the next middleware component.
  • The Run method adds a terminal middleware delegate to the application pipeline. It is always placed at the end of a specific pipeline chain.
  • The Map method is used to branch the middleware pipeline conditionally based on the URL path matching a specific segment.
